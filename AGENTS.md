# AGENTS.md

Guidance for AI coding agents working in the NativeLink repository. Human contributors should read [CONTRIBUTING.md](CONTRIBUTING.md) — this file distills the parts an agent needs and adds the conventions that are common to get wrong.

## What this project is

NativeLink is a high-performance remote build cache and remote execution (RBE) platform implementing the Remote Execution API (REAPI) over gRPC. It is a Rust workspace (edition 2024, MSRV `1.93.1`, current version `1.6.0`), built with Bazel (the source of truth for CI), Cargo, and a Nix dev shell. See [llms.txt](web/apps/docs/public/llms.txt) for a full product/docs map and https://docs.nativelink.com for documentation.

## Repository layout

- `src/bin/nativelink.rs` — the main binary entrypoint.
- `nativelink-config/` — JSON5 config schema (`stores`, `schedulers`, `workers`, `servers`). Example configs live in `nativelink-config/examples/`.
- `nativelink-store/` — storage backends and composable decorator stores (the largest crate).
- `nativelink-scheduler/` — matches actions to workers, tracks execution state.
- `nativelink-worker/` — sandboxed local/remote action execution.
- `nativelink-service/` — gRPC REAPI services (CAS, AC, Execution, ByteStream, Capabilities, Worker API, Health).
- `nativelink-proto/` — protobuf-generated interfaces.
- `nativelink-util/` — shared plumbing: hashing, retries, metrics, the `Store` trait, and the `task` module (see the critical rule below).
- `nativelink-error/`, `nativelink-metric/`, `nativelink-macro/` — cross-cutting support crates.
- `integration_tests/`, `run_integration_tests.sh` — end-to-end tests against a running instance.
- `deployment-examples/`, `kubernetes/`, `local-remote-execution/` — deployment and LRE tooling.
- `web/apps/docs/` — the Fumadocs + Next.js documentation site (managed with `bun`).

## Environment setup

Almost all tooling ships in the Nix flake. Enter the hermetic dev shell before doing anything:

```bash
nix develop        # or use direnv: `direnv allow` once, then it's automatic
```

This is the environment CI uses. Working outside it is possible for pure-Cargo edits but not recommended.

## Build, test, and lint

Prefer Bazel — it mirrors CI. Cargo works for a faster inner loop on pure-Rust changes.

```bash
# Build
bazel build //:nativelink          # or: cargo build

# Test
bazel test //...                   # full suite (also runs rustfmt checks)
bazel test doctests                # documentation tests
cargo test                         # faster Rust-only loop

# Run locally against an example config
nix run . -- ./nativelink-config/examples/basic_cas.json5

# Integration (end-to-end) tests
./run_integration_tests.sh
```

Formatting and linting — run before finishing any change:

```bash
bazel run --config=rustfmt @rules_rust//:rustfmt   # auto-format Rust (bazel test enforces this)
pre-commit run -a                                  # all hooks; applies fixes but does NOT stage them — `git add` after
vale <file>                                         # prose style (docs); errors block, warnings are OK
```

When possible, verify a change end-to-end (build + relevant test target) rather than relying on inspection alone. Report test results honestly, including failures and skips.

## Critical code conventions

- **Never call `tokio::spawn`, `tokio::task::spawn`, `std::thread::spawn`, `Runtime::block_on`, `block_in_place`, or raw runtime builders.** These are `clippy` `disallowed-methods` (see [clippy.toml](clippy.toml)). Use the wrappers in `nativelink-util::task` (`task::spawn`, `background_spawn`, `spawn_blocking`) instead. This is the single most common way an otherwise-correct change fails CI.
- **License header:** every source file starts with the standard NativeLink Authors / Functional Source License header. Copy it from any existing file (e.g. [nativelink-store/src/redis_store.rs](nativelink-store/src/redis_store.rs)) when creating a new one.
- **Imports:** `rustfmt` is configured with `group_imports = "StdExternalCrate"` and `imports_granularity = "Module"` (see [.rustfmt.toml](.rustfmt.toml)). Let the `rustfmt` target handle ordering rather than hand-formatting.
- **Config compatibility:** the JSON5 config is a public compatibility surface. Changing it means updating `nativelink-config`, the examples in `nativelink-config/examples/`, and the autogenerated reference. Prefer additive, backward-compatible changes; call out any breaking change explicitly.
- **Match surrounding code:** mirror the naming, comment density, and idioms of the file you are editing.

## Commit & PR conventions

- Branch off `main`; do not commit directly to it. (Human workflow uses a fork + `upstream` remote; see CONTRIBUTING.md.)
- Commit title: capitalized, imperative mood, **no trailing period**, as short as possible. If you want the word "and" in the title, split the commit. Body wraps at 72 columns.
  - Good: `Add Oracle object store backend`
  - Bad: `Adds oracle store.` / `Add store and fix scheduler bug`
- Commits and tags are expected to be signed.
- To revise an open PR, use `git commit --amend` + `git push -f` rather than stacking `fixup` commits.
- Only commit or push when the user asks.

## Documentation site

The docs live in `web/apps/docs/` (Fumadocs, run on port 3001). Content is MDX under `content/docs/`; some pages are generated:

```bash
cd web
bun --filter @nativelink/docs dev                    # preview
bun --filter @nativelink/docs gen:changelog          # regenerate changelog page from CHANGELOG.md
bun --filter @nativelink/docs gen:config-reference vX.Y.Z   # regenerate config reference for a tag
```

- `web/apps/docs/public/llms.txt` is the served `llms.txt` for LLM agents — keep it in sync when docs structure or product framing changes.
- Do not hand-edit generated files (`content/docs/reference/changelog.md`, `.../nativelink-config/*`); regenerate them.

## Do / don't

- **Do** stay inside `nix develop`, run `rustfmt` + `pre-commit run -a` before wrapping up, and prefer Bazel targets that match CI.
- **Do** use `nativelink-util::task` for all spawning/blocking.
- **Don't** introduce new config fields without updating examples and the reference.
- **Don't** delete or overwrite files you didn't create without surfacing what you found first.
- **Don't** perform outward-facing or hard-to-reverse actions (pushing, publishing, releasing) unless explicitly asked.
